# Web App Plan — Bonus A

Planning doc only — no visual/color/component decisions here on purpose,
that's explicitly what s is doing separately. This defines what the app
needs to DO (flow, pages, data, API shape) so any AI tool (this one,
Antigravity, etc.) builds toward the same target instead of improvising
structure independently. Read PROJECT_CONTEXT.md first — this plan
assumes and does not re-litigate anything in it.

## 0. What this is (and isn't)

The core deliverable is already done and working: `main.py` + `src/` is
a CLI engine that parses a folder of resumes and scores them against a
JD, verified against real data (PROJECT_CONTEXT.md Section 11). Bonus A
is a web front end **on top of** that engine, not a rewrite of it. The
web layer's only job is to (1) collect inputs a browser user can
provide that a CLI user currently provides as folder paths/flags, (2)
call the exact same `src/` functions `main.py` already calls, and (3)
render the exact same data `sample_output.json` /
`parse_quality_report.md` already contain. If a web-layer bug ever
looks like a parsing/scoring bug, it's almost certainly a web-layer bug
— that logic is already tested.

## 1. Recommended stack (minimal, matches the project's existing philosophy)

- **Backend: Flask.** Not FastAPI — Flask's mental model (a function per
  route, `request.files`, `jsonify`) is closer to what's already
  understandable in this codebase, and matches the "stdlib-first,
  minimal deps, one thing to explain per file" pattern used everywhere
  else (PROJECT_CONTEXT.md Section 2). Add `flask` and `werkzeug` to
  `requirements.txt` — that's the entire new dependency footprint.
- **Frontend: plain HTML + CSS + a small amount of vanilla JS
  (`fetch`), no framework.** No React/Vue build step to debug under
  time pressure, no npm install as a new failure point on demo day. s's
  color/component work is CSS on top of plain HTML — nothing here
  blocks that. If more interactivity is wanted later (e.g. live
  sortable tables), that's an additive enhancement, not a rewrite.
- **No database.** Everything is request-scoped: a user uploads
  resumes, the app parses+scores them, returns JSON, done. No
  persistence needed for the hackathon scope. (If asked for "save past
  runs" later, that's a real feature add, not implied by anything here.)

## 2. The actual user flow (this is the part that matters most)

```
1. Landing / Setup page
   - Upload resumes: multi-file picker, .pdf only (matches main.py's
     glob("*.pdf") behavior — reject non-PDFs client-side AND
     server-side, don't trust the client)
   - Choose a JD, one of three ways (mirrors jd_loader.py's 3 input
     paths exactly — don't invent a 4th):
       a) Dropdown of the 5 known roles (data/jds.json)
       b) Paste/upload structured JD JSON
       c) Paste free-text JD (a real pasted job posting) -- this is
          Bonus C, already working (PROJECT_CONTEXT.md Section 11)
   - "Run" button

2. Processing state (while the batch runs)
   - This can genuinely take a while for real data (Section 11: ~1-20s
     per resume depending on provider/model, all sequential LLM calls
     today). Show real progress, not a fake spinner:
       - Minimum viable: a simple "Parsing resume 4 of 12..." counter,
         backend returns progress via polling a /api/status/<job_id>
         endpoint every ~1s.
       - Nicer, if time allows: Server-Sent Events (SSE) streaming each
         resume's parse_status as it completes -- turns OCR/column-split
         firing live into an actual demo moment for judges, instead of
         a report they have to be told about afterward.
   - Whichever you build, the per-resume status shown here must be the
     SAME real fields main.py already produces (parse_method,
     parse_status) -- don't invent new status labels for the UI.

3. Results page
   - Three sections per the existing output shape (Section 9 of
     PROJECT_CONTEXT.md) -- Shortlist / Reserve / Needs Review. Don't
     redesign this grouping, it's already the graded structure.
   - Each candidate row: rank, name, score, confidence badge
     (High/Medium/Low/N/A), parse_quality badge (Clean/Partial/Failed),
     and the exactly-3 reasoning bullets already generated by
     scorer.py. Clicking/expanding a row can show more (skills matched
     list, projects) but the 3 bullets are the primary explainability
     surface -- keep them prominent, they're the "explainable" half of
     "ranked, scored, explainable shortlist" from the problem statement.
   - A visible link/tab to the full Parse Quality Report (every resume,
     independent of scoring, Clean/Partial/Failed + notes) -- the
     problem statement treats this as a first-class deliverable, not a
     debug page, so it shouldn't be buried.
   - If multiple JDs were run (matches --all-jds), tab/switch between
     JD results on this same page rather than a full page reload per JD.

4. (Optional, low priority) History / past runs
   - Not required by anything in PROJECT_CONTEXT.md or the problem
     statement. Skip unless everything above is done with time to
     spare.
```

## 3. Backend API surface (what routes to build)

Keep this thin -- every route is a thin wrapper calling existing
`src/` functions, not new logic:

| Route | Method | Does |
|---|---|---|
| `/api/jds` | GET | Returns the 5 known JDs from `jd_loader.all_known_jd_keys()` + `jd_loader.load()`, for the dropdown |
| `/api/jd/parse-text` | POST | Body: raw JD text. Calls `jd_loader.load(text)` (routes through free-text LLM extraction if needed), returns the structured JD it produced -- lets the frontend show "here's what we understood from your posting" before running the full batch |
| `/api/run` | POST | Multipart: resume files + a `jd` field (role key, JSON, or free text). Saves uploads to a temp dir (see Section 5, security), calls `parse_one_resume` per file + `score_candidate` + `build_shortlist` exactly like `main.py`, returns `job_id` immediately, runs the batch in a background thread |
| `/api/status/<job_id>` | GET | Returns progress: which resumes are done, their parse_status, whether the whole job is finished |
| `/api/result/<job_id>` | GET | Returns the final JSON, same shape as `sample_output.json` |
| `/api/result/<job_id>/report` | GET | Returns the parse quality report (reuse `write_parse_quality_report`'s data, either as JSON for the frontend to render or the raw markdown) |

Everything these routes do is already implemented in `src/` and
`main.py` — the routes just need to call it with web-sourced inputs
(uploaded file paths, form data) instead of CLI args, and hold results
in memory (a dict keyed by `job_id`) instead of writing straight to
`output/`.

## 4. Data contract (reuse, don't reinvent)

The frontend should consume the exact same JSON shape already defined
in PROJECT_CONTEXT.md Section 9 (`sample_output.json`) and the
`ParsedResume`/`ScoreResult` dataclasses in `src/models.py`. Do not
design a new response schema for the web API — serialize the existing
dataclasses (`dataclasses.asdict()`, same as `main.py` already does)
and hand that straight to the frontend. This keeps the web layer
provably consistent with the CLI/report output judges will also see,
which matters if a judge compares the two.

## 5. Security, since this is now a network-facing surface (new, per PROJECT_CONTEXT.md Section 11.7)

The CLI never had to think about this — it only reads a local folder
the operator already controls. A web upload endpoint does:

- **Never trust the client filename.** Use `werkzeug.utils.secure_filename`
  or generate your own filename (e.g. a uuid) for saved uploads —
  don't build a filesystem path directly from what the browser sends.
- **Validate file type server-side, not just via the `<input accept>`
  attribute** (which is a UI hint, not a security control). Check the
  extension AND that PyMuPDF can actually open it as a PDF before
  processing — `pdf_reader.extract()` already handles a bad/corrupted
  PDF gracefully (confirmed in Section 11.5), but reject obviously
  wrong file types before they even reach that code.
- **Set a max upload size** (Flask's `MAX_CONTENT_LENGTH`) and a max
  file count per request — an unbounded multi-file upload is a resource
  exhaustion vector, and each resume triggers a real LLM call that
  costs quota/money.
- **Clean up temp files after each job** — don't accumulate uploaded
  resumes (which contain real PII) on disk indefinitely.
- **Don't return stack traces or internal file paths to the browser**
  on error — catch exceptions at the route level and return a generic
  error message; log the real detail server-side only.
- **CORS**: if frontend and backend are served from the same Flask app
  (recommended — simplest), this doesn't matter. Only relevant if the
  frontend is hosted separately.
- The API-key-leak and PII-redaction fixes from Section 11.7 already
  protect the LLM-call path regardless of whether the input came from
  the CLI or a web upload — nothing new needed there.

## 6. What NOT to build (scope discipline, time is limited)

- No user accounts/auth — this is a single-operator demo tool for the
  hackathon, not a multi-tenant product.
- No database/persistence across runs.
- No async job queue (Celery/Redis/etc.) — a plain Python background
  thread per job is enough at this scale (a handful of resumes per demo
  run, not thousands).
- No new scoring/parsing logic in the web layer, ever. If something
  looks wrong in the UI, the fix belongs in `src/`, not in a route
  handler re-deriving it differently.

## 7. Suggested build order

1. `/api/jds` + a static JD dropdown page — smallest possible slice,
   proves Flask serving works.
2. `/api/run` (synchronous first, no job_id/polling yet) against 1-2
   uploaded resumes + a known JD — proves the upload -> parse -> score
   -> JSON path end to end.
3. Results page rendering that JSON (Shortlist/Reserve/Needs Review +
   reasoning bullets).
4. Free-text JD input wired to `/api/jd/parse-text`.
5. Background job + progress polling (or SSE) — upgrade from
   synchronous once the happy path works, so a slow real batch doesn't
   just hang the browser with no feedback.
6. Parse quality report view.
7. Only then: visual polish (s's colors/components pass).

Steps 5-7 are explicitly gated on 1-4 actually working end-to-end
first — don't build progress UI for a pipeline that isn't wired up yet.
